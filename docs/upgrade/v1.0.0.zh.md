# FastGPT Plugin v1.0.0 更新文档

发布时间：2026-06-01

v1.0.0 是 FastGPT Plugin 的系统工具基础设施升级版本。这个版本把系统工具从仓库内置源码和旧运行时缓存机制，迁移为独立插件包的上传、安装、版本管理和 local-pool 运行时注册机制。

模型配置和 workflow 模板保持兼容，升级时不用做专项迁移。

## 主要变化

### 系统工具插件化

系统工具从旧的 `modules/tool/packages` 内置目录迁移为独立插件包：

- 支持单个 `.pkg` 插件包。
- 支持包含多个 `.pkg` 的官方插件 `.zip` 包。
- 支持上传后确认、URL 安装、版本列表、删除和 disabled 清理。
- 支持基于 `pluginId + version + etag` 的唯一版本识别。
- 服务启动时自动注册 active 插件到 local-pool 运行时。

升级后，用户需要使用官方系统工具 zip 包重新导入系统工具。官方 zip 包由发布方提供，仓库内不维护该压缩包。

### 架构重组

服务从旧 `runtime` 结构迁移为 pnpm workspace monorepo：

- `apps/server`：HTTP 服务入口和路由组装。
- `packages/domain`：实体、值对象和端口定义。
- `packages/usecase`：插件、工具、模型和 runtime 用例。
- `packages/interface-adapter`：API contract 和 DTO。
- `packages/infrastructure`：Mongo、S3、Redis、插件仓储、运行时驱动和静态数据。
- `sdk/client`：新版调用端 SDK。
- `sdk/factory`：系统工具开发 SDK。
- `apps/cli`：系统工具创建、构建、检查、打包 CLI。

### 运行时升级

系统工具运行迁移到 local-pool：

- 每个插件按版本注册为运行服务。
- 支持进程池、队列、超时、重试退避和运行指标。
- 支持插件级 runtime config。
- 插件运行入口从对象存储拉取，并缓存到本地文件目录。

## 兼容性说明

- 模型配置兼容：无需迁移脚本。
- Workflow 模板兼容：无需迁移脚本。
- 旧工具调用方需要继续传 `pluginId`；需要固定版本时可传 `version`。
- 工具详情接口支持 `fallbackLatestVersion=true`，用于请求版本缺失时回退到最新版本。
- 旧 `system_plugins` 中只有 `toolId` 的记录不能直接作为新版插件运行。
- 生产升级建议连接新的 MongoDB 数据库，保留旧库数据用于回滚和审计。

## 数据库变化

### `system_plugins`

v1.0.0 继续使用 `system_plugins` collection，但文档结构已经变更。

旧结构：

```js
{
  type: 'tool',
  toolId: 'getTime',
  objectName: 'optional deprecated value'
}
```

新结构：

```js
{
  pluginId: 'getTime',
  version: '1.0.0',
  etag: 'content-etag',
  type: 'tool',
  name: { en: 'Get Time', 'zh-CN': '获取时间' },
  icon: 'https://...',
  description: { en: '...', 'zh-CN': '...' },
  tags: ['tools'],
  data: {
    toolDescription: '...',
    inputSchema: {},
    outputSchema: {},
    secretSchema: {},
    children: []
  },
  status: 'active',
  createAt: Date,
  updateAt: Date
}
```

索引变化：

```js
// 旧版
{ type: 1, toolId: 1 } // unique

// v1.0.0
{ pluginId: 1, version: 1, etag: 1 } // unique
{ expiredAt: 1 } // TTL
```

升级说明：

- 这里只说明表结构变化。
- 推荐为 v1.0.0 配置新的 MongoDB 数据库。
- 在新库中使用官方插件 zip 包重新导入系统工具，让服务生成完整插件记录。

### `plugin_installations`

v1.0.0 新增 collection，用来记录某个来源安装的插件版本。

```js
{
  source: 'system',
  pluginId: 'getTime',
  version: '1.0.0',
  etag: 'content-etag',
  pluginObjectId: ObjectId('...')
}
```

索引：

```js
{ source: 1, pluginId: 1, version: 1 } // unique
```

这个 collection 由插件上传确认或 URL 安装流程自动生成，升级时无需手工写入。

### `plugin_runtime_configs`

v1.0.0 新增 collection，用来保存插件运行时配置。

```js
{
  pluginId: 'getTime',
  config: {},
  updatedAt: Date
}
```

没有记录时，服务使用默认 local-pool 配置。升级时无需迁移旧数据。

### `s3_ttls`

`s3_ttls` collection 保持同名同结构：

```js
{
  bucketName: 'fastgpt-private',
  minioKey: '...',
  expiredTime: Date
}
```

无需迁移脚本，可以保留旧数据。

## 推荐升级方式

### 1. 停止旧服务

停止旧版 FastGPT Plugin 服务，避免升级期间继续写入旧格式系统工具记录。

### 2. 保留旧数据库

保留旧 MongoDB 数据库和旧对象存储数据。旧库作为回滚和审计来源，升级流程不需要清理旧 `system_plugins` 记录。

生产环境推荐为 v1.0.0 准备一个新的 MongoDB 数据库，并把新版服务的 `MONGODB_URI` 指向新库。例如：

```bash
MONGODB_URI="mongodb://username:password@mongo:27017/fastgpt-plugin-v1?authSource=admin"
```

旧对象存储数据也可以保留，重点包括：

```text
system/plugin/tools
system/plugin/files
```

### 3. 部署 v1.0.0 服务

v1.0.0 使用 Node 22 + pnpm workspace 构建。Docker 镜像运行入口为：

```bash
node dist/main.js
```

生产环境必须设置强 `AUTH_TOKEN`，长度至少 32 位，并避免使用默认值。

### 4. 导入官方系统工具 zip

上传官方插件 zip：

```bash
curl -X POST "$FASTGPT_PLUGIN_BASE_URL/api/plugin/upload" \
  -H "Authorization: Bearer $AUTH_TOKEN" \
  -F "files=@official-system-plugins.zip"
```

记录返回结果中的每个插件唯一标识：

```json
{
  "pluginId": "getTime",
  "version": "1.0.0",
  "etag": "..."
}
```

确认插件：

```bash
curl -X POST "$FASTGPT_PLUGIN_BASE_URL/api/plugin/confirm" \
  -H "Authorization: Bearer $AUTH_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "uniqueIds": [
      {
        "pluginId": "getTime",
        "version": "1.0.0",
        "etag": "..."
      }
    ]
  }'
```

zip 内包含多个插件时，把上传响应中所有 `plugins[].pluginId/version/etag` 都放入 `uniqueIds`。

### 5. 验证升级结果

验证服务健康：

```bash
curl "$FASTGPT_PLUGIN_BASE_URL/health"
```

验证工具列表：

```bash
curl "$FASTGPT_PLUGIN_BASE_URL/api/tools" \
  -H "Authorization: Bearer $AUTH_TOKEN"
```

验证单个工具：

```bash
curl "$FASTGPT_PLUGIN_BASE_URL/api/tool?pluginId=getTime&fallbackLatestVersion=true" \
  -H "Authorization: Bearer $AUTH_TOKEN"
```

验证新 MongoDB：

```js
db.system_plugins.countDocuments({ pluginId: { $exists: true }, status: 'active' })
db.plugin_installations.countDocuments({ source: 'system' })
db.plugin_runtime_configs.countDocuments()
db.s3_ttls.getIndexes()
```

`plugin_runtime_configs` 为 0 属于正常状态。

服务日志中应看到：

```text
Register active plugins on init completed
```

## 重要环境变量

| 变量 | 说明 | 默认值 |
| --- | --- | --- |
| `AUTH_TOKEN` | API 鉴权 token，生产环境必须设置强 token | development 下为 `token` |
| `DISABLE_SSRF_CHECK` | 是否关闭插件 URL 安装安全检查 | `false` |
| `ALLOWED_INSTALL_HOSTS` | 允许 URL 安装访问的远程主机，逗号分隔 | 空 |
| `PLUGIN_RUNTIME_MODE` | 插件运行时模式 | `localPool` |
| `LOCAL_FILE_BASE_PATH` | 插件运行本地缓存目录 | 系统 tmp 下 `fastgpt-plugin` |
| `S3_FILE_BASE_PATH` | 对象存储基础路径 | `system/plugin` |
| `POOL_MAX_TOTAL_PODS` | local-pool 全局最大进程数 | `100` |
| `POOL_SERVICE_MIN_PODS` | 单插件默认最小进程数 | `0` |
| `POOL_SERVICE_MAX_PODS` | 单插件默认最大进程数 | `5` |
| `POOL_SERVICE_IDLE_TIMEOUT` | 空闲进程回收时间，毫秒 | `60000` |
| `POOL_SERVICE_POD_TIMEOUT` | 插件进程执行超时，毫秒 | `120000` |
| `POOL_SERVICE_STARTUP_RETRY_BASE_DELAY` | 插件启动失败重试基础退避，毫秒 | `1000` |
| `POOL_SERVICE_STARTUP_RETRY_MAX_DELAY` | 插件启动失败重试最大退避，毫秒 | `10000` |

`CHECK_INTERNAL_IP` 已由插件安装 URL SSRF 检查配置承接。生产环境建议保持 `DISABLE_SSRF_CHECK=false`，通过 `ALLOWED_INSTALL_HOSTS` 管理允许安装源。

## 对象存储路径变化

旧版系统工具路径：

```text
system/plugin/tools/<toolId>.js
system/plugin/files/...
```

v1.0.0 插件路径：

```text
system/plugin/<pluginId>/<version>/<etag>/index.js
system/plugin/<pluginId>/<version>/<etag>/README.md
system/plugin/<pluginId>/<version>/<etag>/<logo-file>
system/plugin/<pluginId>/<version>/<etag>/assets/<asset-file>
system/plugin/temp/<pluginId>/<version>/<etag>/...
```

旧对象存储文件无需转换，保留旧路径用于回滚和审计。释放存储空间可以另行制定下线清理计划，不作为 v1.0.0 升级步骤。

## 回滚

1. 停止 v1.0.0 服务。
2. 将服务配置切回旧版 MongoDB 和旧对象存储配置。
3. 恢复旧镜像。
4. 启动旧版服务。

已经写入 v1.0.0 新路径的对象存储文件可以保留；确认旧版本不再回滚后，再另行制定下线清理计划。

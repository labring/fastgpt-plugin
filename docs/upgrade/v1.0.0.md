# FastGPT Plugin v1.0.0 Upgrade Guide

Language: [简体中文](./v1.0.0.zh.md) | [English](./v1.0.0.md)

Release date: 2026-06-01

v1.0.0 is an infrastructure upgrade for FastGPT Plugin system tools. This version migrates system tools from repository-built-in source code and the old runtime cache mechanism to independent plugin package upload, installation, version management, and local-pool runtime registration.

Model configuration and workflow templates remain compatible, so no dedicated migration is needed for them during upgrade.

## Major Changes

### System Tools Become Plugins

System tools move from the old built-in `modules/tool/packages` directory to independent plugin packages:

- Supports single `.pkg` plugin packages.
- Supports official plugin `.zip` packages containing multiple `.pkg` files.
- Supports upload confirmation, URL installation, version list, deletion, and disabled cleanup.
- Supports unique version identification by `pluginId + version + etag`.
- Automatically registers active plugins to the local-pool runtime when the service starts.

After upgrade, users need to re-import system tools with the official system tool zip package. The current branch provides `.github/assets/upgrade_pkg.zip` as the upgrade import package. For official releases, use the zip provided as a release asset or inside the deployment package.

### Architecture Reorganization

The service migrates from the old `runtime` structure to a pnpm workspace monorepo:

- `apps/server`: HTTP service entrypoint and route assembly.
- `packages/domain`: entities, value objects, and port definitions.
- `packages/usecase`: plugin, tool, model, and runtime use cases.
- `packages/interface-adapter`: API contracts and DTOs.
- `packages/infrastructure`: Mongo, S3, Redis, plugin repository, runtime drivers, and static data.
- `sdk/client`: new caller-side SDK.
- `sdk/factory`: system tool development SDK.
- `apps/cli`: CLI for system tool creation, build, check, and packaging.

### Runtime Upgrade

System tool execution moves to local-pool:

- Each plugin is registered as a runtime service by version.
- Supports process pool, queue, timeout, retry backoff, and runtime metrics.
- Supports plugin-level runtime config.
- Plugin runtime entry is fetched from object storage and cached to a local file directory.

## Compatibility Notes

- Model configuration is compatible: no migration script is needed.
- Workflow templates are compatible: no migration script is needed.
- Old tool callers should continue to pass `pluginId`; pass `version` when a fixed version is needed.
- The tool detail API supports `fallbackLatestVersion=true`, used to fall back to the latest version when the requested version is missing.
- Old `system_plugins` records containing only `toolId` cannot be run directly as new plugins.
- Production upgrades are recommended to connect to a new MongoDB database and keep old database data for rollback and audit.

## Database Changes

### `system_plugins`

v1.0.0 continues to use the `system_plugins` collection, but the document structure has changed.

Old structure:

```js
{
  type: 'tool',
  toolId: 'getTime',
  objectName: 'optional deprecated value'
}
```

New structure:

```js
{
  pluginId: 'getTime',
  version: '1.0.0',
  etag: 'content-etag',
  type: 'tool',
  name: { en: 'Get Time', 'zh-CN': '获取时间' },
  icon: 'https://...',
  description: { en: '...', 'zh-CN': '...' },
  tags: ['tools'],
  data: {
    toolDescription: '...',
    inputSchema: {},
    outputSchema: {},
    secretSchema: {},
    children: []
  },
  status: 'active',
  createAt: Date,
  updateAt: Date
}
```

Index changes:

```js
// old version
{ type: 1, toolId: 1 } // unique

// v1.0.0
{ pluginId: 1, version: 1, etag: 1 } // unique
{ expiredAt: 1 } // TTL
```

Upgrade notes:

- This section only describes table structure changes.
- Use a new MongoDB database for v1.0.0 when possible.
- In the new database, re-import system tools with the official plugin zip package so the service generates complete plugin records.

### `plugin_installations`

v1.0.0 adds this collection to record plugin versions installed from a source.

```js
{
  source: 'system',
  pluginId: 'getTime',
  version: '1.0.0',
  etag: 'content-etag',
  pluginObjectId: ObjectId('...')
}
```

Index:

```js
{ source: 1, pluginId: 1, version: 1 } // unique
```

This collection is generated automatically by plugin upload confirmation or URL installation flows. No manual write is needed during upgrade.

### `plugin_runtime_configs`

v1.0.0 adds this collection to save plugin runtime configuration.

```js
{
  pluginId: 'getTime',
  config: {},
  updatedAt: Date
}
```

When no record exists, the service uses default local-pool configuration. No old data migration is needed during upgrade.

### `s3_ttls`

The `s3_ttls` collection keeps the same name and structure:

```js
{
  bucketName: 'fastgpt-private',
  minioKey: '...',
  expiredTime: Date
}
```

No migration script is needed, and old data can be kept.

## Recommended Upgrade Flow

### 1. Stop The Old Service

Stop the old FastGPT Plugin service to avoid continuing to write old-format system tool records during the upgrade.

### 2. Keep The Old Database

Keep the old MongoDB database and old object storage data. The old database is used for rollback and audit. The upgrade flow does not need to clean old `system_plugins` records.

For production, prepare a new MongoDB database for v1.0.0 and point the new service `MONGODB_URI` to the new database. For example:

```bash
MONGODB_URI="mongodb://username:password@mongo:27017/fastgpt-plugin-v1?authSource=admin"
```

Old object storage data can also be kept, especially:

```text
system/plugin/tools
system/plugin/files
```

### 3. Deploy The v1.0.0 Service

v1.0.0 uses Node 22 and pnpm workspace for build. The Docker image entrypoint is:

```bash
node dist/main.js
```

Production must set a strong `AUTH_TOKEN` with at least 32 characters and avoid default values.

### 4. Import The Official System Tool Zip

Upload the official plugin zip:

```bash
curl -X POST "$FASTGPT_PLUGIN_BASE_URL/api/plugin/upload" \
  -H "Authorization: Bearer $AUTH_TOKEN" \
  -F "files=@.github/assets/upgrade_pkg.zip"
```

Record the unique identifier of each plugin in the response:

```json
{
  "pluginId": "getTime",
  "version": "1.0.0",
  "etag": "..."
}
```

Confirm plugins:

```bash
curl -X POST "$FASTGPT_PLUGIN_BASE_URL/api/plugin/confirm" \
  -H "Authorization: Bearer $AUTH_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "uniqueIds": [
      {
        "pluginId": "getTime",
        "version": "1.0.0",
        "etag": "..."
      }
    ]
  }'
```

When the zip contains multiple plugins, put all `plugins[].pluginId/version/etag` values from the upload response into `uniqueIds`.

### 5. Verify Upgrade Result

Verify service health:

```bash
curl "$FASTGPT_PLUGIN_BASE_URL/health"
```

Verify tool list:

```bash
curl "$FASTGPT_PLUGIN_BASE_URL/api/tools" \
  -H "Authorization: Bearer $AUTH_TOKEN"
```

Verify a single tool:

```bash
curl "$FASTGPT_PLUGIN_BASE_URL/api/tool?pluginId=getTime&fallbackLatestVersion=true" \
  -H "Authorization: Bearer $AUTH_TOKEN"
```

Verify the new MongoDB:

```js
db.system_plugins.countDocuments({ pluginId: { $exists: true }, status: 'active' })
db.plugin_installations.countDocuments({ source: 'system' })
db.plugin_runtime_configs.countDocuments()
db.s3_ttls.getIndexes()
```

It is normal for `plugin_runtime_configs` to be 0.

Service logs should contain:

```text
Register active plugins on init completed
```

## Important Environment Variables

| Variable | Description | Default |
| --- | --- | --- |
| `AUTH_TOKEN` | API auth token. Production must set a strong token. | `token` in development |
| `DISABLE_SSRF_CHECK` | Whether to disable plugin URL installation security checks | `false` |
| `ALLOWED_INSTALL_HOSTS` | Remote hosts allowed for URL installation, comma-separated | Empty |
| `PLUGIN_RUNTIME_MODE` | Plugin runtime mode | `localPool` |
| `LOCAL_FILE_BASE_PATH` | Local cache directory for plugin runtime files | `fastgpt-plugin` under system tmp |
| `S3_FILE_BASE_PATH` | Object storage base path | `system/plugin` |
| `POOL_MAX_TOTAL_PODS` | local-pool global max process count | `100` |
| `POOL_SERVICE_MIN_PODS` | Default minimum process count for one plugin | `0` |
| `POOL_SERVICE_MAX_PODS` | Default maximum process count for one plugin | `5` |
| `POOL_SERVICE_IDLE_TIMEOUT` | Idle process recycle time, in milliseconds | `60000` |
| `POOL_SERVICE_POD_TIMEOUT` | Plugin process execution timeout, in milliseconds | `120000` |
| `POOL_SERVICE_STARTUP_RETRY_BASE_DELAY` | Plugin startup failure retry base backoff, in milliseconds | `1000` |
| `POOL_SERVICE_STARTUP_RETRY_MAX_DELAY` | Plugin startup failure retry max backoff, in milliseconds | `10000` |

`CHECK_INTERNAL_IP` is now covered by plugin installation URL SSRF check configuration. Production should keep `DISABLE_SSRF_CHECK=false` and manage allowed install sources through `ALLOWED_INSTALL_HOSTS`.

## Object Storage Path Changes

Old system tool paths:

```text
system/plugin/tools/<toolId>.js
system/plugin/files/...
```

v1.0.0 plugin paths:

```text
system/plugin/<pluginId>/<version>/<etag>/index.js
system/plugin/<pluginId>/<version>/<etag>/README.md
system/plugin/<pluginId>/<version>/<etag>/<logo-file>
system/plugin/<pluginId>/<version>/<etag>/assets/<asset-file>
system/plugin/temp/<pluginId>/<version>/<etag>/...
```

Old object storage files do not need conversion. Keep old paths for rollback and audit. Storage cleanup can be planned separately and is not part of the v1.0.0 upgrade steps.

## Rollback

1. Stop the v1.0.0 service.
2. Switch service configuration back to the old MongoDB and old object storage configuration.
3. Restore the old image.
4. Start the old service.

Object storage files written to v1.0.0 new paths can be kept. After confirming the old version will no longer be used for rollback, plan offline cleanup separately.

## Release Checklist

- Old MongoDB database is kept.
- Old system tool object storage paths are kept.
- v1.0.0 service is connected to the new MongoDB database.
- v1.0.0 service is deployed and a strong `AUTH_TOKEN` is set.
- Official system tool zip is uploaded and confirmed.
- `/health` returns normally.
- `/api/tools` returns the system tool list.
- `system_plugins` contains active plugin records in the new structure.
- `plugin_installations` contains installation records with `source: 'system'`.
- Service logs contain `Register active plugins on init completed`.
